1. What Perpensa is

Perpensa is an AI-orchestrated deliberation and research platform. It takes a question - "Should we acquire TargetCo at 9x EBITDA?", "Should we overhaul this turbine or run it to failure?", "Is this market entry defensible?" - and subjects it to structured adversarial reasoning before producing a decision artifact backed by frozen, traceable evidence.

The core workflow follows four phases: Research → Discuss → Challenge → Evolve. AI agents with distinct perspectives (advocate, skeptic, red team) are assembled into teams, debate the question using real evidence, and produce artifacts that capture not just the conclusion but the full reasoning lineage - who argued what, what evidence supported each position, where the team agreed, and where they didn't.

The name comes from the Latin perpēnsa - "having been thoroughly weighed." The platform exists because consequential decisions deserve more than a single analyst's perspective and more than a chatbot's confident answer. They deserve structured opposition, evidence provenance, and a permanent record of how the conclusion was reached.

3. Workflow compositions

Workflows are composable and non-sequential. Any workflow can invoke Create to produce an artifact. Any workflow can call Research at any step. Workflows can spawn other workflows mid-execution. A user can go straight from Research to Create without ever entering a deliberation, or run Attack as a standalone stress test and produce a red team brief directly.

4. Agent catalog

Agents are AI personas with defined roles, perspectives, and behavioral constraints. They are assembled dynamically into teams by the orchestration engine based on the decision question and workflow requirements. New agents can be added without modifying workflows - the agent pool is the extensibility point for vertical specialization.

The Synthesizer deserves special attention. It can be loaded from a prompt template (the default - a general-purpose synthesis prompt that adapts to input type) or created as a specialized custom synthesizer (e.g., a "PE Deal Synthesizer" that knows IC memo conventions, or a "Capital Assessment Synthesizer" that understands asset lifecycle terminology). Either approach works; the choice is made at team assembly time based on the decision domain.

5. Agent API & extensibility

The agent pool is not a closed system. Perpensa exposes an internal Agent API that allows users and partners to create, test, and deploy custom agents. This is the developer platform layer - it turns Perpensa from a product into an ecosystem.

Agent creation paths

There are two paths to creating a custom agent, both producing the same deployable artifact:

Prompt-template path. The simplest route. A user defines an agent's role, perspective, behavioral constraints, and domain context via a structured prompt template in Template Studio. The platform wraps the template in the standard agent interface (input schema, output schema, workflow hooks) automatically. This is how most domain experts and specialized analysts will be created - a PE firm's "regulatory specialist" is a prompt template with FTC merger review context, not custom code.

Code path. For agents that need custom logic - external API calls, proprietary models, complex scoring algorithms, or integration with enterprise systems - users can write agent code directly. The platform provides a chatbot interface that generates LangGraph agent code from natural language descriptions, scaffolding the correct input/output contracts and workflow hooks. Users can also load their own code directly.

Validation & governance

Whether an agent arrives via prompt template or code, it passes through a validation pipeline before deployment:

Behavioral validation. The agent is tested against a suite of scenarios to verify it behaves within its declared constraints - a skeptic agent shouldn't advocate, a domain expert shouldn't opine outside its specialty. Agents that fail behavioral validation are flagged for review, not silently deployed.

Interface compliance. Every agent must implement the standard input/output contract: it receives a deliberation context (the question, current state, evidence available, other agents' positions) and returns a structured response (its position, supporting evidence citations, confidence level, and any requests for further research). Code-path agents that don't conform to this contract are rejected with specific errors.

Sandboxing. Code-path agents run in isolated execution environments. They cannot access other tenants' data, cannot make unauthorized external calls, and cannot modify the deliberation state outside their declared output contract. The validation layer inspects code for prohibited patterns before deployment.

8. Vertical overlays

The base platform handles generic deliberation. Vertical overlays specialize it for specific domains by adding custom agents (via the Agent API), domain-specific templates, and pre-configured workflow patterns.

PE due diligence (Perpensa Capital - Deals)

A PE firm evaluating an acquisition uses Deliberate with advocate and skeptic agents supplemented by a PE analyst, a regulatory specialist, and a market comparables researcher. The Attack workflow runs a red team against the investment thesis. Scenario branches the analysis at different multiples. Create produces a deal memo using the IC deck template - or Create can produce a standalone competitive landscape report directly from Research without any deliberation at all. The Research Vault provides the audit trail that the IC requires.

Capital asset lifecycle (Perpensa Capital - Assets)

A manufacturer managing a fleet of capital assets uses Index to maintain the asset registry, Monitor to watch for lifecycle events (maintenance thresholds, regulatory deadlines, condition changes), Deliberate with maintenance engineer and capital planner agents to evaluate overhaul vs. replace decisions, and Scenario to model different capital planning approaches. Create produces capital plans and asset assessment reports. Custom agents built via the Agent API can integrate directly with enterprise CMMS systems to pull real-time asset condition data into the deliberation.

9. Technology considerations

Language split. The primary question is TypeScript vs. Python for the capture engine and orchestration layer. TypeScript offers strong typing, excellent async patterns for agent orchestration, and a unified stack with the frontend. Python offers a richer ML/NLP ecosystem and more mature libraries for document parsing and vector operations. The current leaning is a TypeScript orchestration layer with Python services for ML-heavy operations, but this decision has not been finalized.

Agent framework. LangGraph is the target framework for agent implementation, providing the graph-based execution model that maps naturally to Perpensa's workflow compositions. The Agent API's chatbot interface generates LangGraph code, and the validation pipeline tests against LangGraph's execution semantics.

Infrastructure stack. Development and early deployment maps across a tiered compute architecture: Synology NAS (storage, data management), Raspberry Pi 5 + Hailo NPU (lightweight inference, edge processing), Jetson Orin Nano (local model serving), and HP Z2/RTX 4070 Ti Super workstation (heavy inference, training, development).

Development tooling. Claude Code with Linear MCP integration and Ralph plugin for agentic development loops. Linear project (Perpensa MVP, ID 249fb60c) is active.

10. Open questions for technical advisors

1. Build vs. integrate on ingestion. How much of the ingestion pipeline (Stage 2 - Parse & Extract) should be built vs. integrated from existing libraries? The document parsing problem is well-solved; the question is whether our normalization and deduplication requirements justify a custom pipeline or whether we compose from existing tools (Unstructured, LlamaParse, Apache Tika) with a thin orchestration layer.

2. Agent API priority. The Agent API is clearly needed, but should it be V1 or V2? A V1 Agent API accelerates the ecosystem and enables partner-built vertical agents, but adds surface area and requires the validation/sandboxing pipeline to be production-ready at launch. A V1 without it means all custom agents are prompt-template-only until V2 ships the code path.

3. Vault implementation. The research vault architecture (content-addressable storage, hash-chain manifest) is well-defined conceptually. The implementation question is whether to build on existing CAS systems (IPFS, git's object store model), adapt a purpose-built store optimized for evidence snapshots, or start with something simpler (S3 + DynamoDB with content hashing) and migrate later. What's the right fidelity for V1 given that the tamper-evidence story is central to the value proposition?

4. Context window management. During a multi-round debate, the accumulated deliberation transcript can easily exceed any model's context window. How do we manage this? Progressive summarization (lossy but predictable), retrieval-augmented context (maintains full fidelity but adds latency), sliding window with full archive access, or a hybrid? The choice affects how "rewind & inject" works - rewinding to an earlier state is trivial with full history but expensive with summarized history.

5. Vector store selection. The semantic index needs to handle evidence embeddings, taxonomy relationships, and similarity search. The options range from embedded (ChromaDB, LanceDB) to managed (Pinecone, Weaviate) to PostgreSQL-native (pgvector). Given that the vault already needs a durable store, does it make sense to co-locate embeddings in PostgreSQL via pgvector, or does the query pattern (high-throughput similarity search during active debate) justify a dedicated vector store?

6. Agent observability. When a deliberation produces a surprising result, someone needs to debug it. What does agent observability look like? Full transcript logging is necessary but not sufficient - you need to see which evidence each agent retrieved, what its confidence was at each turn, and where the debate turned. Is this a custom tracing implementation, or can we build on existing LLM observability frameworks (LangSmith, Arize, Helicone) and extend them for multi-agent orchestration?

7. Testing strategy for prompt templates. A prompt template change can fundamentally alter an agent's behavior. How do we regression-test prompt changes before publishing? The options include golden-set evaluation (run the updated agent against known scenarios and compare outputs), behavioral assertions (the skeptic must challenge at least N claims per round), and A/B deployment (run old and new templates in parallel on the same deliberation). What combination gives us confidence without creating a testing bottleneck?

8. TypeScript/Python split. The current leaning is TypeScript for orchestration and frontend, Python for ML-heavy services (embedding, classification, NLP). Where exactly does the boundary sit? Does the LangGraph agent runtime live in Python (natural fit) or TypeScript (unified with orchestration)? What's the IPC mechanism between the two - HTTP, gRPC, message queue?
